# CPIMS Execution Roadmap
## Multi-Agent, Artifact-Driven Delivery Plan

This roadmap governs CPIMS as a truth-governed, demo-first execution program.

Every phase must follow:

1. truth definition
2. controlled lane execution
3. browser/runtime verification

---

## 1. Roadmap Principles

- business truth is central
- AI is advisory unless approved otherwise
- no fake metrics
- verification is mandatory
- demo clarity outweighs technical novelty

---

## 2. Roadmap Structure

The roadmap is executed in phases. Each phase must define:
- goal
- scope
- entry gate
- pass 1 truth work
- pass 2 build work
- pass 3 verification work
- exit gate

---

## 3. Phase A — Truth Foundation

### Goal
Establish the baseline truth, lifecycle, seeded records, and execution governance.

### Scope
- truth registry
- lifecycle contract
- seeded scenario definitions
- metric definitions baseline
- governance docs
- artifact expectations

### Entry Gate
- project mission approved
- operating charter approved

### Pass 1 — Truth
Define:
- schedule truth
- lifecycle truth
- public/internal boundary
- baseline metrics
- notification truth
- AI labeling truth

### Pass 2 — Build
- governance docs scaffolding
- seeded store alignment
- issue/project board structure
- demo narrative baseline

### Pass 3 — Verify
- review for internal consistency
- verify seeded truth aligns with UI assumptions
- verify no contradictions across docs

### Exit Gate
- truth registry established and accepted
- governance model is operational

---

## 4. Phase B — Core Experience Build

### Goal
Deliver the commissioner-ready citizen and operational prototype path.

### Scope
- homepage
- today’s rule
- schedule
- guidance
- request submission
- acknowledgement
- tracking
- walk-in path
- notice banner

### Entry Gate
- truth registry stable
- lifecycle stable
- seeded data available

### Pass 1 — Truth
Confirm:
- citizen-facing copy
- request flow rules
- walk-in vs appointment vs project review distinctions
- public-safe status visibility

### Pass 2 — Build
- Lane B: citizen-facing experience
- Lane C: minimal supporting staff surfaces if needed
- Lane D: baseline dashboard skeleton

### Pass 3 — Verify
- citizen journey works end to end
- today’s rule is unmistakably clear
- walk-in path does not create artificial barriers
- dashboard reflects seeded truth

### Exit Gate
- citizen-facing golden path works reliably

### Goal
Add operational handling discipline and executive visibility.

### Scope
- staff triage console
- request detail
- status actions
- slot assignment
- notes
- dashboard KPIs
- recent requests
- category visibility

### Entry Gate
- citizen flow stable
- lifecycle transitions proven
- metric definitions approved

### Pass 1 — Truth
Confirm:
- valid staff actions
- note boundaries
- slot assignment semantics
- KPI meanings

### Pass 2 — Build
- Lane C: staff workflow
- Lane D: dashboard
- Lane B: citizen reflections of staff actions

### Pass 3 — Verify
- staff actions correctly update citizen tracker
- dashboard counts derive from same records
- internal notes never leak publicly

---

## 6. Phase 2 — Communication Workflow Layer

### Goal
Model trustworthy, structured citizen communication without depending yet on real external gateway deployment.

### Entry Gate
- lifecycle model stable
- tracking page stable
- staff actions stable
- seeded audit events stable
- dashboard metrics derived from shared truth

### Pass 1 — Truth
Define:
- notification event types
- channel model
- delivery state model
- citizen-visible communication history rules
- dashboard KPI definitions for communications

### Pass 2 — Build
- Lane B: communication history on tracker
- Lane C: notify controls and communication intent actions
- Lane D: communication readiness and delivery-status KPIs

### Pass 3 — Verify
- staff action creates correct communication event
- citizen tracker reflects it
- dashboard counts update from same truth
- no fake `sent` state without modeled event evidence

### Exit Gate
- notification event model implemented
- citizen tracker shows notification history correctly
- staff can trigger notification-intent actions
- dashboard reflects communication KPIs
- communication history feels real and operationally useful

---

## 7. Phase 3 — AI Assistance Layer

### Goal
Model credible AI-assisted capabilities that feel integrated into workflow, not appended for show.

### Entry Gate
- Phase 2 exit gate passed
- AI surfaces defined in truth registry
- advisory boundaries approved

### Pass 1 — Truth
Define:
- AI payload contract
- advisory/override model
- staff visibility rules
- citizen visibility rules
- confidence/provenance labels if needed

### Pass 2 — Build
- Lane B: voice-intake-ready UI and multilingual assistance surfaces
- Lane C: AI summary, suggested priority, suggested routing
- Lane D: AI-usage and AI-assist summary metrics

### Pass 3 — Verify
- AI surfaces feel organic
- advisory nature is clear
- staff control remains obvious
- no UI confusion between human decision and AI suggestion

### Exit Gate
- AI appears as a serious assistive layer, not a decorative layer

---

## 8. Phase 4 — Administrative Intelligence Layer

### Goal
Turn operational records into leadership-grade insight.

### Entry Gate
- Phase 3 exit gate passed
- seeded historical truth available
- taxonomy and tagging truth defined

### Pass 1 — Truth
Define:
- historical data shape
- aging formulas
- zone/category taxonomy
- recurrence logic
- duplicate signal model

### Pass 2 — Build
- Lane C: advanced lookup and historical context
- Lane D: trend charts, aging buckets, cluster views, zone-wise demand patterns
- Lane B: minimal or no major change

### Pass 3 — Verify
- all analytics trace back to data truth
- dashboard remains understandable
- leadership insights are immediate, not overcomplicated

### Exit Gate
- dashboard moves from status visibility to decision-support value

---
